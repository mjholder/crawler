## Session 2 — Architecture refactor (loose coupling)

### What I worked on

Reviewed all scripts for OOP violations and refactored the attack system to use signals throughout, eliminating every case where an object was passed across ownership boundaries.

**Problem identified:** Three separate places called `target.take_damage()` directly via `has_method()` duck typing — in `player._do_attack()`, `enemy._perform_action()`, and `skeleton._perform_action()`. `event.gd` also stored a `var player` reference and took a `Player` argument in `start()`, spreading player ownership beyond `game.gd`.

**Changes made:**

- `enemy.gd` — added `signal attacked(damage: float)`, removed `target` param from `take_turn()` and `_perform_action()`, base `_perform_action()` emits `attacked` instead of calling `take_damage()`
- `skeleton.gd` — `_perform_action()` override now calls `super._perform_action()` then SFX + animation; duck typing removed
- `player.gd` — added `signal attacked(damage: float)`, `_do_attack()` emits signal, `execute_action()` no longer takes a `target` parameter
- `event.gd` — removed `var player`, removed player parameter from `start()`, no combat-specific signals or methods on the base class
- `combat_event.gd` — implemented from stub: enemy roster via `add_enemy()`, forwards enemy `attacked` → `player_attacked`, `receive_player_attack()` routes damage to first live enemy
- `game.gd` — removed stale TODO comment, fixed `start_event(event: Event)` type, type-switch (`if event is CombatEvent`) connects/disconnects combat-specific signals in `start_event` and `_on_event_complete`; `game.gd` is now the sole class that touches `player`

**Signal chains established:**
- Enemy attacks player: `enemy.attacked` → `CombatEvent._on_enemy_attacked` → `player_attacked` → `game._on_player_attacked` → `player.take_damage()`
- Player attacks enemy: `player.attacked` → `game` (via combat type-switch) → `CombatEvent.receive_player_attack` → `enemy.take_damage()`

### Outcomes

#### Completed
- All `has_method("take_damage")` duck typing eliminated
- Player reference now lives only in `game.gd` — no event or enemy class holds one
- `CombatEvent` signal chain implemented (roster management + two-way damage routing)
- Documented in design.md — 2026-03-01

#### Incomplete / deferred
- `CombatEvent` turn loop (`_on_running` stub) — driving `enemy.take_turn()`, detecting wave completion, transitioning to RESOLUTION still to do

### Key decisions made

- Signal-based attacks with player reference isolation. See design.md — 2026-03-01.
- `player_attacked` and `receive_player_attack` live only on `CombatEvent`, not the base `Event` class. `game.gd` uses `if event is CombatEvent` to wire them — keeps the base class clean, easy to extend with new event types.
- `execute_action()` no longer accepts a target — action callables signal their intent, the event layer decides what gets hit.
