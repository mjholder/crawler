## What I worked on

Updated `scripts/game.gd` to introduce the event layer. Added `var current_event: Event`,
a `start_event(event)` entry point that connects `event_complete` and calls `event.start(player)`,
and an `_on_event_complete()` handler that clears the reference and resets state to `NO_TURN`.
Removed the old `load_*_event()` mocks and the `enemies` array тАФ those now belong to the event
that spawns them. The turn flow stubs in `game.gd` are kept with TODOs flagging that `CombatEvent`
will eventually drive them.

Implemented `scripts/event.gd` as the base class for all event types. It owns the phase enum
(`SETUP тЖТ RUNNING тЖТ RESOLUTION тЖТ COMPLETE`), the `event_complete` signal, and the phase
transition logic. Subclasses override the three virtual hooks (`_on_setup`, `_on_running`,
`_on_resolution`) and call `_advance_phase()` from `_on_running()` when they are done. The
base class drives SETUP тЖТ RUNNING automatically on `start()`; RESOLUTION тЖТ COMPLETE fires
when `_advance_phase()` is called.

Also hit and resolved a `class_name` indexing lag in the Godot editor тАФ `Event` wasn't
recognised as a type in `game.gd` until focus returned to the editor and triggered a rescan.

## Key decisions made

- Base class auto-advances SETUP тЖТ RUNNING (synchronous init); RUNNING is the async pause point
- Subclasses call `_advance_phase()` to proceed тАФ base class never auto-advances out of RUNNING
- `_set_phase()` is the internal dispatcher; subclasses read `phase` but never write it
- `game.gd` stays blind to phase internals тАФ it only sees `start()` and `event_complete`
- See design.md тАФ 2026-02-22 (Event state machine entry covers the rationale)
