## What I worked on

Planned and implemented the full GUI layer per the gui-design.md spec. This included a `GUI` CanvasLayer script with MainMenu, PauseMenu, and CombatHUD, all fully wired to game logic via API methods rather than direct node references.

Key implementation work:

- **`scripts/gui.gd`** — New full implementation (`class_name GUI extends CanvasLayer`). Owns all UI state: shows/hides menus, manages a `_enemy_bars` dictionary keyed by Enemy node, instantiates enemy health bars from `_health_bar_scene` export. Public API: `show_main_menu`, `start_game`, `handle_esc`, `return_to_main_menu`, `show/hide_combat_hud`, `update_player_health`, `add/remove/update_enemy_health_bar`, `set_player_turn`, `log_message`.
- **`scenes/game.tscn`** — Restructured GUI node tree: added MainMenu (Title, StartButton, QuitButton) and PauseMenu (Overlay, ResumeButton, QuitToMainButton) subtrees; renamed `PlayerHealth` → `PlayerHealthLabel`; added `PlayerHealthBar` ProgressBar; moved AttackButton into ActionMenu; moved CombatLog into CombatHUD; added gui.gd script and health_bar.tscn export on the GUI node.
- **`scripts/game.gd`** — Wired `@onready var _gui: GUI`; moved debug combat launch from `_ready()` into `_on_start_button_pressed()`; `set_player()` now connects `player.damaged` and initialises the health display; per-enemy health bar add/update/remove wiring in `start_event()`; replaced all direct `$GUI/CombatHUD.show/hide()` calls with GUI API; added `_on_player_damaged`, `_on_quit_to_main`, ESC handling in `_unhandled_input`; `set_player_turn` and `log_message` calls in turn flow.
- **`scenes/skeleton.tscn`** — Removed the in-scene HealthBar node (GUI owns bars now).

## Key decisions made

- GUI owns enemy health bars, not enemy scenes — bars are instantiated from a `_health_bar_scene` export on the GUI node and tracked in a Dictionary keyed by Enemy reference. Enemies emit `damaged`/`died` signals; game.gd connects lambdas that call GUI API.
- `game.gd` calls GUI through its API only — the only direct child-node access from game.gd into the GUI subtree is wiring the two button signals in `_ready()`.
- Debug combat launch moved to `_on_start_button_pressed()` so it goes through the normal game-start flow.
