## What I worked on

### Design / ideation
Spent time in `ideas.md` roughing out three interconnected systems before writing any more code, to make sure the architecture would support the long-term vision:

- **Grid-Based Equipment System** — body-grid inventory (RE/Diablo-style applied to body slots); armor, clothing, weapons, shields, jewelry; modifier surface area for attack, defense, passives, encumbrance, set bonuses, and sockets
- **Core Player Stats** — seven stats: Health (derived), Defense, STR, CON, AGI, SPI, LCK; weapon class gating per stat; Spirit framed as occult/devotional power from an unknown higher being (not scholarly magic)
- **Roguelike Run Structure** — run-based loop, multi-floor descent, escalating difficulty, themed floors; four encounter types: combat, skill checks, loot, roleplay events; design principle to keep systems theme-agnostic and data-driven

### Code
All three scripts touched:

**`player.gd`** — replaced the single hardcoded `attack()` method with an action registry. Actions are `Callable`s stored in a dictionary; `register_action(name, callable)` adds them, `execute_action(name, target)` calls them and emits `turn_ended` exactly once. `_do_attack` is the first registered action. New actions (spells, items, etc.) are a single `register_action` call with no changes to `game.gd`.

**`enemy.gd`** — no changes needed. The existing `_perform_action(target)` override hook already gives each enemy type full control over its own AI decision making, while `game.gd` only ever calls `take_turn(player)`.

**`game.gd`** — full rewrite from stub:
- `TurnState` enum: `PLAYER_TURN`, `ENEMY_TURN`, `GAME_OVER`
- `round_number` counter increments at the start of each player turn
- `set_player(p)` — explicit setup instead of scene-tree scanning; connects player signals; safe to call on save load
- `load_combat_event(enemy_list)` — called by a future `CombatEvent`; registers enemies, connects their signals, starts the turn loop
- `load_skill_check_event()`, `load_loot_event()`, `load_roleplay_event()` — mocked stubs for the other encounter types; TODOs explain their intended contracts
- Turn flow: player acts → `turn_ended` → enemy loop → back to player; player or all-enemies-dead transitions to `GAME_OVER`

## Key decisions made

- Action registry over hardcoded methods — `execute_action` is the single turn-end guarantee — see design.md — 2026-02-22
- Enemy AI fully encapsulated in `_perform_action` override — game.gd is enemy-type-agnostic — see design.md — 2026-02-22
- Explicit `set_player` / `load_*_event` over `_collect_participants` — supports save loading and event-driven encounter flow — see design.md — 2026-02-22
- Spirit (SPI) chosen as the magic stat over Intelligence — occult/devotional flavour, power from an unknown higher being

## Blockers / struggles

- Stats are still flat `@export` vars (`attack_damage`, `max_health`); they'll need to be replaced with the full stat system before weapon gating and modifiers can work
- Action names are plain strings — no compile-time safety; worth defining constants once the action list grows
